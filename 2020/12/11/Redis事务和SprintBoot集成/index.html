事务

==redis的命令保证原子性但是事务不保证原子性==

Redis 事务可以一次执行多个命令， 并且带有以下三个重要的保证：

  • 批量操作在发送 EXEC 命令前被放入队列缓存（顺序性）。
  • 收到 EXEC 命令后进入事务执行，事务中任意命令执行失败，其余的命令依然被执行（一次性）。
  • 在事务执行过程，其他客户端提交的命令请求不会插入到事务执行命令序列中（排他性）。

一个事务从开始到执行会经历以下三个阶段：

  • 开始事务(multi)

  • 命令入队(其他命令)。

  • 执行事务(exec)。

    ==所有的命令在事务中，并没有直接被执行！只有发起执行命令才会执行 EXEC==

放弃事务，discard

命令 作用
multi 开启事务
exec 执行事务
discard 放弃事务

异常的种类

  • 编译异常，就是代码错误，没有这命令，这个事务会全部失败
  • 运行时异常，语法没有错误，但是执行时出错，除了有异常的语句，其他的都可以继续执行

redis实现乐观锁

[ WATCH key key …] 监视一个(或多个) key ，如果在事务执行之前这个(或这些) key 被其他命令所改动，那么事务将被打断。

类似于我们的version一样，在监视的时候就会获得当前的值，如果后面执行事务的时候，这个值被修改了，那么整个事务就会失败。如果修改失败，使用 unwatch取消监视，然后再使用watch获得最新的数据，在执行事务
